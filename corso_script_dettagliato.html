Cos'è Claude e cos'è Claude Code

Prima cosa: chiarisci la differenza tra il modello e l'agente. Claude è il modello AI di Anthropic — il cervello. È quello che capisce, ragiona, scrive. Lo trovi su claude.ai, dentro le API, dentro Claude Code, dentro Cursor — è sempre lo stesso modello. È il motore intellettuale.

Perché uno e non l'altro? Claude in chat è perfetto per ragionare, fare brainstorming, scrivere copy a freddo — quando vuoi solo testo. Claude Code lo usi quando devi agire sul sistema: leggere i tuoi prodotti Shopify, comprimere video, mandare un'email, lanciare uno scraper. Il 90% del lavoro in azienda è di questo secondo tipo.

Da dire: "Claude pensa. Claude Code agisce. Per il lavoro vero in azienda serve quello che agisce."

06

MCP — i ponti verso sistemi esterni

Cos'è in semplice: MCP (Model Context Protocol) è un plugin che dà a Claude le credenziali e i comandi per parlare con un'app esterna. Pensa a un adattatore già pronto. Tu lo attivi una volta, e da quel momento Claude può "chiamare" quell'app come se fosse un suo tool interno — leggere ordini Shopify, scrivere su Drive, mandare un messaggio Telegram, leggere insights Meta.

Quando usarlo: quando per quell'app esiste già un MCP pronto e ti serve un'azione strutturata e ripetibile (creare un prodotto, leggere ROAS, salvare un file). È la mano più veloce e affidabile, perché parla la lingua dell'API ufficiale.

Esempio live: lancia una query a Shopify via MCP, vedi gli ultimi 5 ordini. Sotto il cofano è Claude che parla con Shopify in nome tuo. Anticipa: oltre a MCP, Claude usa anche la shell (CLI) e il browser come faresti tu — sono le altre due "mani" verso il mondo.

Da dire: "MCP = ponte pronto. Quando c'è, lo usi sempre — è la via più veloce e pulita per fare parlare Claude con un sistema esterno."
07

CLI — la shell del Mac, il backstop universale

Cos'è in semplice. CLI vuol dire Command Line Interface — la riga di comando. Su Mac è zsh (su Linux/WSL è bash). Sono i comandi che scriveresti tu a mano nel terminale: ls, cd, git commit, ffmpeg, python script.py. Claude Code non li simula: prende il tuo terminale e ci scrive dentro direttamente. È la stessa shell che useresti tu, con la differenza che digita lui.

Differenza chiave con MCP. L'MCP è un plugin specifico per un'app specifica (Shopify MCP, Meta MCP, Drive MCP) — qualcuno l'ha scritto, configurato, mantenuto. La CLI invece è universale: qualunque tool open-source o commerciale installabile sul Mac diventa subito utilizzabile da Claude, senza che nessuno scriva un MCP per quel tool. Per questo è il backstop universale: quando non c'è un MCP — e nel 90% dei casi minori non c'è — c'è quasi sempre un comando CLI che fa la stessa cosa.

Le 5 famiglie che uso ogni giorno (fai vedere a schermo o leggi):

1. Video / audio → ffmpeg (comprime/converte/taglia video, gira sui PDP Shopify), whisper (trascrive audio in testo), yt-dlp (scarica qualsiasi video da YouTube/TikTok/IG con un URL).

2. Codice / deploy → git + gh (commit, branch, pull request da terminale), shopify CLI (deploy del tema senza aprire l'admin), vercel/netlify per il frontend.

3. Web / API → curl (chiama qualunque API REST), jq (parsa il JSON di risposta, prende solo quello che serve), wget (scarica file in batch).

4. Dati / file → python per qualsiasi script, sed/grep/awk per cercare e sostituire testo in centinaia di file in 1 secondo.

5. Sistema / Mac → brew per installare nuovi tool (brew install <nome> è il "ti serve quel comando? eccolo"), open per aprire app/file, osascript per pilotare AppleScript.

Esempio live (fai vedere). Apri terminale, prendi un video raw.mov da 1 GB. Chiedi a Claude: "comprimi questo per Shopify, target 80 MB, mantieni qualità". Claude scrive ffmpeg -i raw.mov -vcodec libx264 -crf 30 -preset slow out.mp4, ti chiede conferma, lancia. Trenta secondi dopo: video da 80 MB pronto. Zero MCP coinvolti. Zero API key. Solo shell + ffmpeg già installato.

La regola pratica che insegno. Prima cerco un MCP per quel servizio (più strutturato, più affidabile). Se non c'è, cerco il tool CLI — quasi sempre esiste. Se non è installato, lo installo con brew install <nome>. Da quel momento è "parte del vocabolario" di Claude per sempre, su quel Mac.

Potenza nascosta che pochi capiscono. L'universo open-source CLI è enorme — decine di migliaia di tool su Homebrew. Sono tutti documentati nel --help, già "lingua nativa" per Claude perché ha letto la loro documentazione in training. Non serve scrivere integrazioni: basta che il comando esista sul Mac e Claude lo usa come se l'avesse usato mille volte. Questo moltiplica la sua capacità all'infinito senza dover scrivere una riga di codice di integrazione.

Da dire: "L'MCP è il plugin pulito quando c'è. La CLI è il backstop universale: tutto quello che gira nel terminale del Mac, Claude lo lancia. Cinque famiglie da ricordare: video, codice, web, dati, sistema. Quando non c'è MCP, c'è quasi sempre un comando shell — e Claude lo conosce già."

Scraper — Apify e Firecrawl per estrarre dati dal web

Per scraping seri (volumi grandi, anti-bot, cookie complessi) due tool dedicati: Apify e Firecrawl. Sono servizi cloud + MCP / SDK. Claude li chiama, loro fanno il lavoro sporco, ti tornano JSON pulito.

Apify: piattaforma con migliaia di "actor" pre-fatti — TikTok scraper, Instagram scraper, Google Maps, LinkedIn, Amazon, Trustpilot, ecc. Ogni actor è uno scraper pronto, paghi a uso. Esempio: voglio i commenti TikTok dei top video competitor → Apify actor TikTok comments → 1000 commenti in 2 minuti.

Firecrawl: scraping di siti web generici trasformati in markdown pulito. Tre comandi: scrape (singola URL), crawl (intero sito), search. Output ottimizzato per LLM. Esempio: voglio leggere tutto il sito di un competitor → firecrawl crawl → ho un solo MD con tutto, Claude lo legge in un colpo.

Differenza con CLI/browser: per 1-2 URL usi browser. Per centinaia/migliaia usi questi servizi. Costano (qualche centesimo per chiamata) ma ti risparmiano ore.

Da dire: "Scraping da hobbista = browser. Scraping serio = Apify o Firecrawl. Costano poco, valgono ore del tuo tempo."

API keys — come si configurano e dove si salvano

"Per ogni servizio esterno (Shopify, Meta, Apify, Firecrawl, OpenAI, ecc.) serve una API key. È la password macchina-a-macchina. Senza, Claude non può fare niente con quel servizio."

Dove prenderle: nel pannello del servizio (es. Shopify Admin → Apps → Create app → genera token). Ogni servizio ha la sua sezione "Developer" o "API". Lì copi la key.

Cosa costruiamo — il contabile che non dorme

Frame: terzo agent del corso, il più "da soldi". Vede gli ordini Shopify, vede le fatture del fornitore (l'invoice coi costi reali), incrocia i due, e ogni giorno: ti manda il report su Telegram o WhatsApp, scrive il CSV del giorno, e archivia tutto in una cartella Drive ordinata. La contabilità giornaliera sparisce dalla tua lista.

Punto chiave: non è un report di "fatturato". È un report di profitto — quanto ti è rimasto in tasca davvero, ogni sera.

Da dire: "Il fatturato è vanità, il profitto è verità. Questo agent vi dice la verità ogni mattina."
02

Le 5 metriche che contano

Definiamo le 5 metriche. Revenue = quanto incassa (lordo). COGS = costo del venduto (quello che paghi al fornitore). GW = Gross profit = revenue − COGS. Net = GW − ads spend − fees. BE ROAS = la soglia di ROAS al punto di breakeven.

Esempio numerico a schermo: revenue €10.000, COGS €3.000, ads €4.000, fees €500. GW = €7.000. Net = €2.500.

Da dire: "Se non sai questi 5 numeri ogni mattina, stai giocando senza vedere lo scoreboard."
03

Hard rules — niente stime, niente overwrite

Le regole non negoziabili che finiranno scritte nella spec. Uno: niente stime — se manca un dato, la casella resta vuota e l'agent lo segnala. Mai inventare metriche. Due: se manca il costo di uno SKU nella fattura fornitore → STOP + notifica. Mai fallback a percentuali o medie.

Tre: niente overwrite di CSV esistenti su Drive o in locale — ogni giorno il suo file, i test in cartella /TEST/ con prefisso TEST_. Questi tre errori sono i killer della fiducia nei dati: uno solo violato e butti via tutto il sistema.

Da dire: "Niente stime, niente fallback, niente overwrite. Tre cose. Il giorno che l'agent inventa un numero, smettete di credergli per sempre."
04

Le due fonti — ordini Shopify + invoice del fornitore

Fonte 1: gli ordini Shopify via API (il token della L7). Dettaglio tecnico che vale oro: filtrare per processed_at (data effettiva del pagamento), non per data di creazione — sennò i numeri non tornano mai con l'admin.

Fonte 2: l'invoice del fornitore — il CSV/XLSX coi costi unitari per SKU. L'agent lo legge, costruisce la tabella SKU → costo, e per ogni riga d'ordine moltiplica quantità × costo. COGS reali, non stimati.

Da dire: "Shopify dice quanto è entrato. La fattura del fornitore dice quanto è uscito. L'agent incrocia i due e ti dice quanto è rimasto."
05

Build con la agent factory — la spec del contabile

Terzo giro di factory, ormai è routine. /agent-spec: "Voglio un agent che ogni giorno: 1) legge gli ordini Shopify del giorno (processed_at), 2) legge i costi dalla fattura fornitore, 3) calcola revenue, COGS, GW, net e BE ROAS, 4) scrive il CSV del giorno, 5) lo archivia su Drive in cartelle anno/mese, 6) mi manda il riassunto su Telegram. Hard rules: niente stime, STOP se manca un costo, mai sovrascrivere file esistenti."

Poi tasks → execute → verify. Sottolinea: le hard rules del punto 03 sono DENTRO la spec — l'agent nasce già con le regole, non gliele ricordi ogni volta.

Da dire: "Le regole si scrivono nella spec, una volta. Da lì in poi l'agent le rispetta sempre. Anche alle 6 di mattina. Anche a Natale."
06

Il report — Telegram o WhatsApp, 30 secondi di lettura

Cos'è un bot Telegram: un contatto Telegram per i programmi — si crea gratis in 2 minuti con BotFather (visto in L5). Il formato del report: data, revenue, ordini, AOV, COGS, GW, ads spend, net, BE ROAS, top 5 prodotti. Emoji semaforo 🟢/🔴 sui KPI. Lettura: 30 secondi.

Alternativa WhatsApp per chi vive lì: stesso messaggio, canale diverso. Il principio non cambia: l'output di un agent deve arrivare dove vivi tu, non in un file che non aprirai mai.

Da dire: "Alle 6 ho letto il giorno prima. Alle 6 e mezzo ho già preso le decisioni. Alle 7 sono operativo."
07

Il CSV giornaliero + la cartella Drive ordinata

Oltre al messaggio, ogni giorno l'agent scrive il CSV: una riga per ordine con tutti i numeri, più la riga dei totali. Naming fisso: YYYY-MM-DD_report.csv. È la tua contabilità giorno per giorno, pronta per il commercialista.

L'archivio su Drive si organizza da solo: Contabilità/2026/06-Giugno/ — l'agent crea le cartelle che mancano e carica il file nel posto giusto. E stampa SEMPRE il link al file in chat: senza link, l'output è invisibile.

Da dire: "Fine anno: 365 CSV, già divisi per mese, già su Drive. Il commercialista vi ama. E voi non avete mai aperto Excel."
08

Schedulazione + log — ogni mattina, e se si rompe lo dice

Sveglia launchd alle 06:00 (il pattern della L5): l'agent parte da solo ogni mattina. E le due regole di affidabilità: ogni run scrive un log (start, end, esito), e se qualcosa va storto — Shopify non risponde, manca la fattura — l'agent NON muore in silenzio: ti manda il messaggio d'errore su Telegram.

Da dire: "Quando un agent fallisce, deve urlare. Mai morire in silenzio. Mai."
09

La prova dal vivo — run completo davanti alla camera

Gran finale: lanci l'agent manualmente sul giorno precedente. Davanti alla camera: legge gli ordini, incrocia i costi, calcola, scrive il CSV, lo carica su Drive (mostra la cartella che si popola), e... notifica sul telefono. Apri il messaggio, leggi i numeri a voce.

Chiusura + ponte: "i numeri arrivano sul telefono ogni mattina. Ma nella prossima lezione li mettiamo anche in diretta, nel browser: la dashboard della contabilità."

Da dire: "Questo report a un contabile costa 200 euro al mese. Il vostro lo fa ogni notte, gratis, senza ferie."

Lezione 10 · 25–30 min

La dashboard dell'agent contabile: i numeri in diretta nel browser

01

Quando serve un frontend

Non sempre. Regola: frontend quando devi consultare dati visivamente o quando il sistema deve essere usato da qualcuno che non sta in terminale (collaboratori, team). Se sei solo tu e sono numeri rapidi → il Telegram della L9 basta.

Per la contabilità il frontend ha senso: vuoi vedere i numeri salire DURANTE il giorno, non solo il riassunto della mattina dopo.

Da dire: "Frontend non è bellezza. Frontend è 'lo guardo dieci volte al giorno' o 'serve a qualcun altro'. La cassa in diretta vale entrambe."
02

Stack minimo — HTML, JS, CSS, niente framework

La dashboard è tre file: una pagina HTML, un file JS, un CSS. Niente React, niente build step. Aprila: è plain HTML, qualsiasi cosa lì dentro Claude la sa scrivere e modificare.

Grafica: SHIN-EN del way-stack (la skill shinen-design) — la dashboard esce già con la faccia giusta, coerente con tutti gli altri tool.

Da dire: "Plain HTML finché regge. Quando non regge più, allora i framework. Mai prima. E con SHIN-EN esce già bella."
03

L'architettura — l'agent serve i dati, la pagina li mostra

Pattern semplice: l'agent contabile della L9 espone un piccolo endpoint JSON sul Mac (i numeri del giorno, sempre freschi). La pagina fa fetch('/api/kpi') ogni 30 secondi e aggiorna i numeri a schermo. L'agent fa da centralina (chiama Shopify lui, al massimo ogni 30s — così non sfori i limiti API), la pagina fa da cruscotto.

Da dire: "Backend = centralina. Frontend = cruscotto. La pagina non parla mai con Shopify direttamente — parla con l'agent."
04

Build live — le card della contabilità

Costruisci la dashboard con Claude davanti alla camera. Le card: revenue del giorno grande in alto (il numero protagonista), poi ordini, AOV, COGS, GW, net. E la riga BE ROAS: soglia di breakeven vs ROAS effettivo — sopra la linea navighi, sotto affoghi.

Trick UX: il contatore animato. Quando arriva un ordine, il numero gira e sale. Effetto cassa di Las Vegas, gratis. Responsive: 4 colonne desktop, 1 colonna mobile — il KPI in tasca tutto il giorno.

Da dire: "Quando arriva un ordine, lo vedo entrare. Il numero che gira è dopamina. La pagina diventa il tuo posto."
05

Deploy — locale gratis, o online per il team

Per te solo: gira sul Mac, accedi da localhost, zero costi. Una sveglia launchd fa partire il server all'avvio = "sempre lì".

Quando deve vederla qualcun altro o tu da fuori casa: Render. Push su GitHub, Web Service dal repo, build automatico, ~$7/mese. Dominio tuo via Cloudflare. Accesso protetto: una password condivisa col team, niente complessità.

Da dire: "Se sei solo tu, locale e gratis. Se serve al team, online col tuo dominio. 5 minuti dal commit alla produzione."
06

Esercizio finale — il sistema contabile completo

Esercizio finale del blocco ecommerce: il sistema completo. Agent contabile (L9) + dashboard (L10): connetti Shopify, configura i costi fornitore, schedula il report delle 06:00, builda la dashboard, deploy.

Tempo target: una giornata la prima volta. Da quel momento: report ogni mattina, CSV ogni giorno su Drive, numeri in diretta nel browser. Quando il team vede gli stessi numeri tuoi, lavora con la tua stessa testa.

Da dire: "Report alle 6, archivio su Drive, cassa in diretta nel browser. Questa è contabilità da dirigente, costruita in due lezioni."

Bonus 01 · 30–35 min

Brain Marketing: il Creative Strategist che scrive le tue ADV

01

Cos'è un brain — il metodo della L6, applicato al marketing

Richiamo: nella L6 abbiamo costruito il knowledge di Hormozi dai transcript. Stesso identico metodo, fonti diverse: stavolta dentro ci vanno i miei documenti di marketing. E in più un pezzo nuovo: il voice contract — il file che dice al brain CHI è, come parla, cosa dice e cosa non dice mai.

Il risultato si chiama brain: una persona finta che vive nel vault. Il nostro: il Creative Strategist — il senior creativo + copywriter direct-response che scrive le ADV per i tuoi brand.

Da dire: "Un brain non è un prompt. È una persona finta che vive nel vault: ha studiato i tuoi materiali, parla con una voce precisa, e non dimentica mai niente."
02

Le fonti — i documenti di marketing da dargli in pasto

4 cartelle da popolare in sources/. brand_voice/ = guidelines di brand, esempi di copy che ami. products/ = schede prodotto con feature, benefit, USP. customer_voice/ = commenti, recensioni, DM, Q&A dei tuoi clienti reali — l'oro vero: le parole ESATTE con cui i clienti descrivono il problema. top_ads/ = le tue ads che hanno funzionato (link + screenshot + perché hanno funzionato).

Regola: solo materiale REALE del brand. Niente cose mai testate. Quello che metti in sources/ è quello che lui ha studiato; quello che non metti, non lo sa.

Da dire: "4 cartelle, materiale vero. Più ce ne metti, più sa il tuo brand a memoria. E il customer_voice è l'oro: le ads migliori sono scritte con le parole dei clienti."
03

Build del brain — sources → lessons → wiki

La pipeline della L6, lanciata sui documenti marketing: vault-ingest spezza le fonti in lessons (una lesson = un concetto), l'index si aggiorna, la wiki di sintesi si genera da sola. 10-15 minuti di run.

Dettaglio in più per il marketing: le lessons si taggano in 4 famiglie — hook (i primi 3 secondi), angle (la storia/promessa), copy (il testo del corpo), offer (lo scambio: prezzo + bonus + garanzia). Quando interroghi puoi pescare solo dalla famiglia che serve.

Da dire: "4 leve: hook, angle, copy, offer. Una è rotta — la campagna è rotta. Il brain le tiene tutte e quattro vive."
04

Il voice contract — chi è il Creative Strategist

Il file CLAUDE.md del brain, sezione per sezione. Voice: senior creative + direct-response copywriter, diretto, zero fuffa. Do: cita sempre la fonte (quale lesson, quale ad di riferimento), propone sempre più varianti. Don't: non inventa numeri di benchmark — se non li ha, dice "non li ho"; non scrive copy generico da brochure.

Esempio do/don't concreto a schermo. Do: "Hook A vince perché il primo frame interrompe il pattern (fonte: lesson 03)." Don't: "Hook A è migliore per varie ragioni." (vago, niente fonte). Senza voice contract risponde Claude generico. Con voice contract risponde il personaggio.

Da dire: "Il Creative Strategist non scrive copy generico. Scrive copy citando la lezione su cui si fonda. Tracciabilità totale."
05

La skill /creative-strategist

La porta d'ingresso: la skill. Mostra la SKILL.md: cosa carica (voice contract + lessons + top ads del brand), come si invoca, output format. Da qui in poi: una slash e sei nella stanza con lui.

Come si compone bene la richiesta: prodotto (cosa), pubblico (a chi), obiettivo (cosa deve fare il copy), constraint (lunghezza, tono). 4 parti = output mirato. Antipattern: "scrivi una pubblicità" — vago = mediocre.

Da dire: "Domanda vaga, risposta vaga. Prodotto, pubblico, obiettivo, constraint — e la risposta è usabile subito."
06

Al lavoro 1 — il copy per le ADV

Prima demo live: /creative-strategist copy per [prodotto], pubblico [X], obiettivo acquisto, formato primary text Meta. Output: 3 varianti di copy, ognuna con razionale e fonte citata. Leggi a voce, commenta cosa terresti e cosa no.

Poi la variante: "riscrivi la versione 2 più corta e più aggressiva" — iterazione in secondi, non in meeting.

Da dire: "Tre copy in 30 secondi, ognuno col perché. Quello che un copywriter ti fattura 500 euro a brief."
07

Al lavoro 2 — gli script delle ADV

Seconda demo: lo script video. /creative-strategist script video 30s per [prodotto], stile UGC. Output: script scena per scena — hook (0-3s), problema (3-8s), meccanismo (8-18s), prova (18-25s), CTA (25-30s) — con indicazioni visive per ogni scena e il parlato completo.

Punto didattico: lo script non è "un testo da leggere". È un documento di produzione: chi gira il video sa esattamente cosa inquadrare e cosa dire, secondo per secondo.

Da dire: "Hook, problema, meccanismo, prova, CTA. Trenta secondi divisi al secondo. Lo passi a chi gira e non ci sono dubbi."
08

Al lavoro 3 — il brief

Terza demo: il brief completo per creator/editor. /creative-strategist brief per batch di 5 video su [prodotto]. Output: per ogni video — angle, hook, scene chiave, copy on-screen, do/don't di brand, riferimenti dalle top ads. Un documento che mandi al creator e torna il girato giusto.

È il documento che chiude il cerchio: il brain non ti dà solo idee — ti dà il pezzo di carta con cui le idee diventano produzione.

Da dire: "Il brief è il ponte tra l'idea e il girato. Cinque video briffati in un comando. Il creator ringrazia, voi risparmiate una giornata."
09

Strutturare ADV vincenti — il loop

Il metodo per renderle vincenti: il brain genera 10 ipotesi (hook + angle), tu ne scegli 2-3 da testare, le mandi in produzione col brief, misuri i risultati, e — pezzo fondamentale — riporti i risultati nel brain: le winner finiscono in top_ads/, le loser in una nota "non funziona". Il prossimo batch parte già più intelligente.

Realismo finale: l'agent non è infallibile. Lui propone, tu approvi. Mai pubblicare a scatola chiusa. Tu sei il direttore creativo; lui è il junior instancabile che non si offende mai.

Da dire: "Ogni batch insegna al prossimo. Velocità di apprendimento composta. E il filtro finale siete sempre voi: lui propone, voi decidete."
