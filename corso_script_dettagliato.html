Cos'è Claude e cos'è Claude Code

Prima cosa: chiarisci la differenza tra il modello e l'agente. Claude è il modello AI di Anthropic — il cervello. È quello che capisce, ragiona, scrive. Lo trovi su claude.ai, dentro le API, dentro Claude Code, dentro Cursor — è sempre lo stesso modello. È il motore intellettuale.

Perché uno e non l'altro? Claude in chat è perfetto per ragionare, fare brainstorming, scrivere copy a freddo — quando vuoi solo testo. Claude Code lo usi quando devi agire sul sistema: leggere i tuoi prodotti Shopify, comprimere video, mandare un'email, lanciare uno scraper. Il 90% del lavoro in azienda è di questo secondo tipo.

06

MCP — i ponti verso sistemi esterni

Cos'è in semplice: MCP (Model Context Protocol) è un plugin che dà a Claude le credenziali e i comandi per parlare con un'app esterna. Pensa a un adattatore già pronto. Tu lo attivi una volta, e da quel momento Claude può "chiamare" quell'app come se fosse un suo tool interno — leggere ordini Shopify, scrivere su Drive, mandare un messaggio Telegram, leggere insights Meta.

Quando usarlo: quando per quell'app esiste già un MCP pronto e ti serve un'azione strutturata e ripetibile (creare un prodotto, leggere ROAS, salvare un file). È la mano più veloce e affidabile, perché parla la lingua dell'API ufficiale.

Esempio live: lancia una query a Shopify via MCP, vedi gli ultimi 5 ordini. Sotto il cofano è Claude che parla con Shopify in nome tuo. Anticipa: oltre a MCP, Claude usa anche la shell (CLI) e il browser come faresti tu — sono le altre due "mani" verso il mondo.

Da dire: "MCP = ponte pronto. Quando c'è, lo usi sempre — è la via più veloce e pulita per fare parlare Claude con un sistema esterno."
07

CLI — la shell del Mac, il backstop universale

Cos'è in semplice. CLI vuol dire Command Line Interface — la riga di comando. Su Mac è zsh (su Linux/WSL è bash). Sono i comandi che scriveresti tu a mano nel terminale: ls, cd, git commit, ffmpeg, python script.py. Claude Code non li simula: prende il tuo terminale e ci scrive dentro direttamente. È la stessa shell che useresti tu, con la differenza che digita lui.

Differenza chiave con MCP. L'MCP è un plugin specifico per un'app specifica (Shopify MCP, Meta MCP, Drive MCP) — qualcuno l'ha scritto, configurato, mantenuto. La CLI invece è universale: qualunque tool open-source o commerciale installabile sul Mac diventa subito utilizzabile da Claude, senza che nessuno scriva un MCP per quel tool. Per questo è il backstop universale: quando non c'è un MCP — e nel 90% dei casi minori non c'è — c'è quasi sempre un comando CLI che fa la stessa cosa.

Le 5 famiglie che uso ogni giorno (fai vedere a schermo o leggi):

1. Video / audio → ffmpeg (comprime/converte/taglia video, gira sui PDP Shopify), whisper (trascrive audio in testo), yt-dlp (scarica qualsiasi video da YouTube/TikTok/IG con un URL).

2. Codice / deploy → git + gh (commit, branch, pull request da terminale), shopify CLI (deploy del tema senza aprire l'admin), vercel/netlify per il frontend.

3. Web / API → curl (chiama qualunque API REST), jq (parsa il JSON di risposta, prende solo quello che serve), wget (scarica file in batch).

4. Dati / file → python per qualsiasi script, sed/grep/awk per cercare e sostituire testo in centinaia di file in 1 secondo.

5. Sistema / Mac → brew per installare nuovi tool (brew install <nome> è il "ti serve quel comando? eccolo"), open per aprire app/file, osascript per pilotare AppleScript.

Esempio live (fai vedere). Apri terminale, prendi un video raw.mov da 1 GB. Chiedi a Claude: "comprimi questo per Shopify, target 80 MB, mantieni qualità". Claude scrive ffmpeg -i raw.mov -vcodec libx264 -crf 30 -preset slow out.mp4, ti chiede conferma, lancia. Trenta secondi dopo: video da 80 MB pronto. Zero MCP coinvolti. Zero API key. Solo shell + ffmpeg già installato.

La regola pratica che insegno. Prima cerco un MCP per quel servizio (più strutturato, più affidabile). Se non c'è, cerco il tool CLI — quasi sempre esiste. Se non è installato, lo installo con brew install <nome>. Da quel momento è "parte del vocabolario" di Claude per sempre, su quel Mac.

Potenza nascosta che pochi capiscono. L'universo open-source CLI è enorme — decine di migliaia di tool su Homebrew. Sono tutti documentati nel --help, già "lingua nativa" per Claude perché ha letto la loro documentazione in training. Non serve scrivere integrazioni: basta che il comando esista sul Mac e Claude lo usa come se l'avesse usato mille volte. Questo moltiplica la sua capacità all'infinito senza dover scrivere una riga di codice di integrazione.

Da dire: "L'MCP è il plugin pulito quando c'è. La CLI è il backstop universale: tutto quello che gira nel terminale del Mac, Claude lo lancia. Cinque famiglie da ricordare: video, codice, web, dati, sistema. Quando non c'è MCP, c'è quasi sempre un comando shell — e Claude lo conosce già."

Scraper — Apify e Firecrawl per estrarre dati dal web

Per scraping seri (volumi grandi, anti-bot, cookie complessi) due tool dedicati: Apify e Firecrawl. Sono servizi cloud + MCP / SDK. Claude li chiama, loro fanno il lavoro sporco, ti tornano JSON pulito.

Apify: piattaforma con migliaia di "actor" pre-fatti — TikTok scraper, Instagram scraper, Google Maps, LinkedIn, Amazon, Trustpilot, ecc. Ogni actor è uno scraper pronto, paghi a uso. Esempio: voglio i commenti TikTok dei top video competitor → Apify actor TikTok comments → 1000 commenti in 2 minuti.

Firecrawl: scraping di siti web generici trasformati in markdown pulito. Tre comandi: scrape (singola URL), crawl (intero sito), search. Output ottimizzato per LLM. Esempio: voglio leggere tutto il sito di un competitor → firecrawl crawl → ho un solo MD con tutto, Claude lo legge in un colpo.

Differenza con CLI/browser: per 1-2 URL usi browser. Per centinaia/migliaia usi questi servizi. Costano (qualche centesimo per chiamata) ma ti risparmiano ore.

Da dire: "Scraping da hobbista = browser. Scraping serio = Apify o Firecrawl. Costano poco, valgono ore del tuo tempo."

API keys — come si configurano e dove si salvano

"Per ogni servizio esterno (Shopify, Meta, Apify, Firecrawl, OpenAI, ecc.) serve una API key. È la password macchina-a-macchina. Senza, Claude non può fare niente con quel servizio."

Dove prenderle: nel pannello del servizio (es. Shopify Admin → Apps → Create app → genera token). Ogni servizio ha la sua sezione "Developer" o "API". Lì copi la key.

Dashboard, deploy e contabilità: dall'agent al report delle 06:00 fino a LiveKPI sul tuo dominio

01

Quando serve un frontend

Non sempre. Regola: frontend quando devi consultare dati visualmente o quando l'agent deve essere usato da qualcuno che non sta in terminale (collaboratori, team). Se sei solo tu e sono numeri rapidi → Telegram basta.

Esempi che chiedono frontend: LiveKPI (cassa live continua), Customer Care Dashboard (collaboratore non-tech che deve approvare), GrowthGuide (loop multi-step con stato).

Stack minimo — HTML, JS, CSS, niente framework

Mostra LiveKPI: una pagina HTML, un file JS, un CSS. Niente React, niente Next, niente build step. Apri il file, è plain HTML.

Da dire: "Plain HTML finché regge. Quando non regge più, allora React. Mai prima."
03

Fetch dati e render

Pattern: l'agent espone un endpoint JSON (FastAPI o Flask sul Mac). Il frontend fa fetch('/api/data') ogni N secondi e aggiorna i numeri nel DOM. Niente WebSocket per cose semplici — polling è abbastanza.

Trick UX: transizioni di numeri (counter animato). Quando il fatturato sale, vedi i numeri girare. Effetto cassa di Las Vegas, gratis.

Da dire: "Polling per i numeri. Il numero che gira è dopamina. La pagina diventa il tuo posto."
04

Deploy — locale o Render

Per dashboard solo tue: gira sul Mac, accedi da localhost, zero costi. Launchd job che fa partire il server all'avvio = "sempre lì".

Quando deve vederla qualcun altro o tu da fuori casa: Render. Push su GitHub, Web Service da repo, env vars, build automatico. Dominio Cloudflare CNAME → Render. Costo: $7/mese a service.

Accesso protetto: cookie session auth semplice, una password condivisa col team. Niente OAuth, niente complessità.

Da dire: "Se sei solo tu, locale e gratis. Se serve al team, Render e dominio. 5 minuti dal commit alla produzione."
05

Contabilità — le 5 metriche che contano

Definiamo le 5 metriche. Revenue = quanto incassa (lordo). COGS = costo del venduto. GW = Gross profit = revenue − COGS. Net = GW − ads spend − fees. BE ROAS = la soglia di ROAS al punto di breakeven.

Esempio numerico: revenue €10.000, COGS €3.000, ads €4.000, fees €500. GW = €7.000. Net = €2.500.

Da dire: "Se non sai questi 5 numeri ogni mattina, stai giocando senza vedere lo scoreboard."
06

Hard rules — no IVA scorporo, no estimate, no overwrite

Tre regole non negoziabili. Uno: niente scorporo IVA nei calcoli di margine ads (revenue lordo Shopify). Due: niente stime — se manca il dato, card vuota. Mai inventare metriche.

Tre: niente overwrite di CSV/XLSX esistenti su Drive o locale. Test re-runs → cartella /TEST/ + prefisso TEST_. Questi tre errori sono i killer della fiducia nei dati: uno solo violato e butti via tutta la dashboard.

Da dire: "Niente scorporo, niente stime, niente overwrite. Tre cose. Non negoziabili."
07

Pull dati — ordini Shopify + COGS reali da XLSX

API Shopify GET /orders.json. Filtri importanti: processed_at (data effettiva pagamento, non created_at), status financial = paid. 30 righe di Python per estrarre tutti gli ordini del giorno.

COGS: il fornitore ti dà un XLSX con costo unitario per SKU. L'agent legge l'XLSX, costruisce dict SKU→cost, per ogni line_item moltiplica qty × cost.

HARD RULE: se manca COGS per uno SKU → STOP + notifica. MAI fallback a matrici % o medie. Dato mancante = card vuota, niente trucchi.

Da dire: "processed_at, non created_at. E niente COGS reale = report bloccato. Mai stimare."
08

Profit e BE ROAS

Formule esplicite. Per ogni ordine: profit = revenue − cogs − shipping − fees. Aggregato giorno = somma dei profit ordine.

BE ROAS = la soglia di ROAS sotto la quale stai bruciando soldi. Mostra in dashboard: BE ROAS oggi a 1.78x, ROAS effettivo a 2.4x. Differenza positiva = sei sopra il break-even.

Da dire: "BE ROAS è la linea di galleggiamento. Sopra = navighi. Sotto = affoghi. Visivamente, una riga sul grafico."
09

Schedule launchd 06:00. L'agent: pull ordini, calcola margini, formatta messaggio Telegram al gruppo del brand. Formato: data, revenue, ordini, AOV, COGS, GW, ads spend, net, ROAS, BE ROAS, top 5 prodotti. Lettura 30 secondi, emoji semaforo 🟢/🔴 sui KPI.

Alert aggiuntivo: se net giornaliero scende sotto soglia → push Telegram immediato alle 23:55. "Il giorno andato male te lo dico io. Non lo scopri a fine settimana."

Da dire: "Alle 6 ho letto il giorno prima. Alle 6 e mezzo ho già preso le decisioni. Alle 7 sono operativo."

Step up: il report arriva alle 06, ma in giornata vuoi vedere i numeri salire in tempo reale. LiveKPI = dashboard browser, polling ogni 30 secondi: revenue, ordini, AOV, BE ROAS aggiornati. Mostra il tuo LiveKPI live.

Architettura: backend FastAPI (porta 8787) che pulla Shopify al massimo ogni 30s e serve da cache (rate limit salvi); frontend che fa fetch /api/kpi e re-renderizza. Backend = centralina, frontend = cruscotto.

Card: revenue grande in alto (numero protagonista), ordini, AOV, BE ROAS. Contatore animato: quando arriva un ordine, lo vedi entrare. Responsive: 4 col desktop, 1 col mobile — il KPI in tasca tutto il giorno.

Da dire: "Il browser sempre aperto su un secondo monitor. Quando arriva un ordine, lo vedo entrare. È la cassa in diretta."

Da dire: "Quando LiveKPI sale per la prima volta sul tuo dominio, sei dall'altra parte. Sei costruttore. Sei dirigente del tuo sistema."
