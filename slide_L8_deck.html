Le operazioni che fai ogni settimana nell'admin — a voce.

Il backend a voce.

01

File Word + immagini → prodotto creato via API. In draft: la pubblicazione la decidi tu.

"Porta la M a 50 pezzi", "alza del 10% tutte le varianti". Operazioni in blocco.

02 · La PDP che converte
構造
Una PDP che converte non è design — è una sequenza di obiezioni gestite.

Anatomia di una PDP.

≈12
Una dozzina di sezioni, ognuna gestisce un'obiezione.Buy box · badges · reviews · benefits · before/after · comparison · FAQ · social proof. Ogni sezione = un file che l'agent sa scrivere.
Blocco 02obiezioni, non decorazioni
03 · La replica
複製
Si replica la struttura — il copy e le foto sono i tuoi.

Replica sezione per sezione.

step 1
Estrai la struttura
Dai l'URL di riferimento → l'agent produce la mappa: sezioni, layout, elementi.
step 2
Costruisci le sezioni
Una alla volta: file Liquid → push sul dev → confronto side-by-side → aggiusti a voce.
regola etica
Struttura sì, copy no
La struttura è informazione, niente copyright. Copy col tuo brand voice, foto tue.
Blocco 03estrai → costruisci → confronta
04 · Modifiche custom
自在
Il tema è una cartella di file: qualsiasi modifica, a voce.

Custom a voce.

stile
Il colore
"Bottone Aggiungi al carrello nero, testo bianco" → trova il file, modifica, preview.
elemento
Un badge
"Badge spedizione gratis sotto il prezzo" → nuovo snippet, incluso.
screenshot
Da immagine
Incolli lo screenshot di un dettaglio → "fammi questo" → lo replica.

Non ti serve sapere il Liquid: ti serve saper descrivere cosa vuoi.

05 · Deploy
公開
La rete di sicurezza: dev sempre, live solo con conferma.

Dev → live.

Dev theme
Sempre.
Ogni push va sul dev theme con l'URL di preview. Nessuno lo vede, sistemi con calma.
Live
Solo con conferma.
Sul live solo con conferma esplicita — è un vincolo della spec. Trucco: dopo il push, un secondo tocco sull'asset busta la cache.
→ clienti vedono in 5s→ non 10 minuti
