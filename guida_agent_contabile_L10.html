01 · Cosa costruiamo

Un agent, due modalità

Non costruiamo due programmi: costruiamo un solo agent contabile che lavora in due modalità, dalle stesse fonti dati e con lo stesso config.json.

Modalità · Live
Dashboard
Web server sempre acceso. Un poller rilegge i dati ogni 120s e li tiene in cache. Espone /api/metrics e una pagina con login: i KPI in diretta nel browser.
Modalità · Report
Giornaliero
Schedulato 1 volta al giorno (16:00). Genera l'Excel a 6 fogli, lo salva per anno/mese, lo carica su Drive e manda il recap su Telegram.
Le fonti dati (cloud, condivise dalle due modalità): Shopify Admin API (ordini, refund, fee, clienti), Triple Whale (spesa ads, ROAS), Google Drive (XLSX costi fornitore), Telegram (notifiche), e — opzionale — Supabase (storico KPI).

03 · config.json

Il cuore di tutto

Un solo config.json serve all'agent in entrambe le modalità. Le sezioni principali:

# config.json — sezioni
shopify        # shop_url, api_key, api_password, api_version
triple_whale   # api_key, shop_url, currency (EUR)
google_drive   # credentials_file, root_folder_id, folder IDs
cogs           # matrix[categoria][paese][quantità] = COGS per ordine
products       # {id_prodotto: nome, label, prezzo, cogs}
finance        # fee gateway, soglie ROAS
exchange_rates # cambi valuta (fallback)
email_watcher  # gmail + app_password (legge il CSV costi)
report         # company_name, file_prefix, timezone
La cogs.matrix è la tabella costi per prodotto/paese/quantità: è la fonte dei COGS quando manca il CSV del fornitore. Va riempita coi tuoi costi reali, 1:1.
04 · Build

Build con la agent factory — la spec

Stesso metodo del corso: la agent factory. Lancia /agent-spec e detta il contratto per intero. Le hard rules vanno dentro la spec: l'agent nasce già con le regole.

# dentro Claude Code:
/agent-spec

# detti, a voce:
"Voglio un agent contabile per il mio store: un solo programma, due modalità.

 MODALITÀ REPORT (1×/giorno, schedulato alle 16:00):
  1) legge gli ordini Shopify del giorno (filtro processed_at, esclude €0)
  2) legge i costi dal CSV del fornitore (o dalla matrix COGS in config)
  3) calcola revenue, COGS, gross profit, net, BE ROAS + ROAS da Triple Whale
  4) genera un Excel a 6 fogli (Analisi, Ordini, Prodotti, Paesi, Customers, Meta)
  5) lo salva in locale per anno/mese, lo carica su Drive, manda il recap su Telegram

 MODALITÀ LIVE (web server sempre acceso):
  - espone /api/metrics con i KPI del giorno + 90 giorni
  - un poller rilegge i dati ogni 120s e li tiene in cache su disco
  - una pagina HTML in stile SHIN-EN con login a cookie

 HARD RULES: niente stime; STOP + notifica Telegram se manca un costo SKU;
 mai sovrascrivere file (test in /TEST/); timezone Europe/Rome; 1 sola istanza."

Poi il giro completo: /agent-tasks → /agent-execute → /agent-verify. La factory costruisce, tu guardi e commenti.

05 · Modalità report

Il flusso di un run

1
COGS del giornoCerca i costi in ordine: CSV del fornitore → matrix in config. Regola dura: se manca il CSV e non c'è la matrix, si ferma.
2
Ordini ShopifyLegge gli ordini del giorno con filtro processed_at (data del pagamento), esclude gli ordini a €0.
3
Calcolo COGSPer ogni riga d'ordine: quantità × costo (per prodotto/paese/quantità). COGS reali, non stimati.
4
Triple Whale + refundSpesa ads (Meta/TikTok/Google), ROAS attribuito e blended, refund del giorno, nuovi vs returning.
5
Genera + distribuisciExcel a 6 fogli → salva in locale (anno/mese) → carica su Drive → recap su Telegram con il link.
Test: lancia l'agent su una data passata in modo test — scrive in /TEST/ e non manda Telegram veri: python main.py 2026-06-16 (o flag LEISHAPE_TEST=1). Verifica che crei l'Excel e lo carichi nella cartella TEST.
06 · Modalità live

La dashboard

Stesso agent, modalità web. Web server Python (stdlib, niente framework), avviato con python server.py sulla porta da PORT (default 8787). Un thread HTTP + un thread poller che ogni 120s rilegge i dati e li tiene in cache in memoria e su disco.

RottaAuthCosa fa
/sì (login)la dashboard HTML coi KPI
/login · /logoutpubblicologin a cookie firmato (user/pass da env)
/api/metricssìJSON con tutti i KPI (oggi + 90 giorni)
/api/refreshsìforza un refresh subito

Priorità delle fonti KPI: se c'è l'XLSX costi del giorno su Drive → è la verità. Se manca (tipico per oggi/ieri) → pipeline live: Shopify + Triple Whale + matrix COGS. Net = vendite − refund − ads − COGS − fee.

07 · Deploy

Schedulazione + deploy online

Modalità report — schedulazione (scegline una): un cron job del cloud (Render Cron / Railway) che lancia python main.py 1×/giorno alle 16:00 — l'opzione più pulita; oppure launchd/cron in locale; oppure un loop scheduler.py come worker sempre acceso.

Modalità live — deploy su Render (o equivalente) come web service:

type: web · runtime python
buildCommand: pip install -r requirements.txt
startCommand: python server.py
healthCheckPath: /
disk: /data 1GB (persistente)
# env: SHOPIFY_*, TRIPLE_WHALE_API_KEY, GOOGLE_SERVICE_ACCOUNT_JSON (inline),
#      LIVEKPI_USER, LIVEKPI_PASS, GDRIVE_*_FOLDER_ID, DATA_DIR=/data, PORT
Cold start: al primo avvio la dashboard scarica lo storico 90 giorni (~10 min, normale). Dopo si aggiorna ogni 2 minuti. Dominio tuo via Cloudflare, accesso protetto da password.
08 · Regole dure

Le regole che fanno la differenza

Dati
Niente stime
Se manca un costo SKU → STOP + notifica Telegram. Mai inventare i COGS, mai fallback a percentuali. È voluto.
File
Mai sovrascrivere
Ogni giorno il suo file. I test vanno in /TEST/ con prefisso TEST_: mai toccare i report veri.
Runtime
TZ + 1 istanza
Timezone Europe/Rome (i confini di giornata dipendono da qui). La modalità live è stateful: 1 sola istanza con disco persistente.

Il take-away

Un solo agent, due modalità: la dashboard live nel browser e il report giornaliero su Drive e Telegram, dalle stesse fonti e dallo stesso config.json. Le credenziali prima, la spec dettata alla factory, le hard rules dentro la spec. Costruito una volta, lavora ogni giorno — anche quando non ci sei.
